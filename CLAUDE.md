# Claude Instructions

## Project Context

This is a **drone racing simulator with a built-in map editor**, built in **Rust** using **Bevy 0.18.0**. See `ARCHITECTURE.md` for the full module structure, data flow, and design decisions.

Key facts:
- **State machine**: `AppState` (Menu → Editor → Race → Results) with `EditorMode` SubStates (ObstacleWorkshop, CourseEditor)
- **Physics**: Thrust-through-body quadrotor with cascaded PID (position → attitude) in `FixedUpdate`, 12 AI drones with per-drone variation. Quadratic drag, angular dynamics with moment of inertia, motor lag (25ms). Aerodynamic perturbations (dirty air, prop wash), battery sag.
- **Data**: Obstacle definitions and courses serialized as RON. Single `.glb` file for all obstacle 3D models, separate `.glb` for drone model.
- **Gate validation**: Trigger volumes (AABB), gates must be passed in order. Hit/miss = crash.

## Performance

The performance target is a stable **60fps**. Performance is paramount.

- If a feature request would risk this target, **flag it immediately** before implementing. Explain specifically why it is a risk (e.g., expensive per-frame system queries, O(n²) algorithms in hot paths, excessive entity/component churn) and propose at least one alternative approach.
- If existing code is encountered that poses a performance risk, flag it proactively even if it wasn't the subject of the current task.
- Prefer solutions with predictable, bounded performance over those with better average-case but worse worst-case behavior.
- Be mindful of Bevy-specific pitfalls: over-querying, large archetypes, and unnecessary system ordering constraints that block parallelism.

## Project Conventions

- **Plugins**: each module exposes a single plugin (struct or fn). Plugins register their own systems with appropriate state guards via `run_if(in_state(...))`.
- **State scoping**: use `DespawnOnExit` on entities that should be cleaned up when leaving a state. Do not manually despawn in `OnExit` unless there's a reason.
- **Serialization**: all persistent data uses RON via serde. Obstacle library lives at `assets/library/default.obstacles.ron`, courses at `assets/courses/*.course.ron`.
- **Asset loading**: all obstacle models come from a single `assets/models/obstacles.glb`. Individual objects are accessed via `Gltf::named_nodes` → `GltfNode` → `GltfMesh` → primitives, using the Blender object name. Each obstacle spawns a parent entity with child `Mesh3d`/`MeshMaterial3d` per primitive.
- **Physics**: all physics systems run in `FixedUpdate` with `.chain()` for ordering. No physics in `Update`.
- **Rendering**: All visible geometry (ground, obstacles, drones) uses `CelMaterial` — a custom material with cel-shading, halftone dot transitions, and hue-shifted highlights/shadows. `CelLightDir` resource stores the world-space light direction, computed once at startup. Use `cel_material_from_color(base_color, light_dir)` to create materials. Explosion particles remain `StandardMaterial` (unlit emissive). Skybox is a `SkyboxMaterial` on an inverted sphere (front-face culled), procedural TRON-style night sky. Custom WGSL shaders live in `assets/shaders/`.
- **Bevy 0.18 specifics**: `set_parent_in_place()` (not `set_parent()`), `SceneRoot` (not `SceneBundle`), `Mesh3d`/`MeshMaterial3d` components, `ChildSpawnerCommands` (not `ChildBuilder`) for `with_children` closures in commands context, `AccumulatedMouseMotion`/`AccumulatedMouseScroll` from `bevy::input::mouse` (not in prelude). `MessageReader<T>` (not `EventReader<T>`) for reading events. `KeyboardInput` from `bevy::input::keyboard`. System tuples max ~12 elements for `run_if`; split larger groups into multiple `add_systems` calls. `Gltf::named_nodes`/`named_scenes` use `Box<str>` keys (not `String`).
- **Menu pattern**: `AvailableCourses` resource is created on `OnEnter(Menu)` and removed on `OnExit(Menu)`. `SelectedCourse` resource persists across states to carry the user's course selection into Race.
- **Workshop pattern**: `WorkshopState` resource is created on `OnEnter(ObstacleWorkshop)` and removed on `OnExit`. Preview entities use `PreviewObstacle` component and are manually despawned on exit (not part of UI hierarchy). glTF node list populates asynchronously once the asset is loaded. If no glb scene matches, a placeholder cube is spawned.
- **Drone pattern**: `DroneGltfHandle` is loaded `OnEnter(Race)`. `DroneAssets` is extracted from the glTF in an `Update` polling system (async pattern). `spawn_drones` polls until both `DroneAssets` and `CourseData` are available, then spawns 12 drones with `DespawnOnExit(AppState::Race)`. Resources are cleaned up `OnExit(Race)`. Path generation functions live in `drone/paths.rs` (`RacePath`, `generate_race_path`, `generate_drone_race_path`, `generate_return_path`, `compute_start_positions`, `adaptive_approach_offset`). Physics runs in `FixedUpdate` as an 11-system `.chain()` using a thrust-through-body model: `update_ai_targets → compute_racing_line → proximity_avoidance → hover_target → position_pid → attitude_controller → dirty_air_perturbation → motor_lag → apply_forces → integrate_motion → clamp_transform`. Key components: `PositionPid` (outer loop, blended derivative: `-velocity + desired_velocity * feedforward_blend`, where `desired_velocity = velocity_hint * max_speed` — at default blend 0.85 the PID maintains speed along the spline instead of braking to a stop near high-curvature gates), `AttitudePd` (inner loop, slightly underdamped for visible settle wobble: kp=7.0, kd=0.20), `DesiredAttitude` (bridge between loops), `DroneDynamics` (velocity, angular velocity, thrust with motor lag 25ms, quadratic drag). Per-drone variation in `DroneConfig` includes PID gains, cornering aggression (±20%), braking distance (±20%), attitude PD multipliers (±10%), `racing_line_bias` (midleg lateral shift, correlated with aggression), `approach_offset_scale` (approach distance multiplier, inversely correlated with aggression), and `gate_pass_offset` (fraction 0.2–0.4 of each gate's trigger volume half-extents, used for 2D pass-through offset in both width and height — aggressive drones use wider offsets). Each drone generates its own unique spline via `generate_drone_race_path()` at spawn time. `RaceSeed` resource (random `u32`, generated fresh each race in `spawn_drones`) is mixed into all Fibonacci hashes in spline generation and return path generation, plus perturbation phase offsets in `dirty_air_perturbation` — this ensures race outcomes vary between runs instead of the same drone indices always getting the same offset directions. Cleaned up `OnExit(Race)`. If the drone glb is missing, a red placeholder cube is used.
- **Course obstacle cleanup**: `spawn_obstacle()` adds `DespawnOnExit(AppState::Race)` to all spawned obstacle entities, ensuring they are automatically cleaned up when leaving Race state. `CourseSpawned` marker resource is removed `OnExit(Race)` to allow re-spawning on re-entry.
- **Drone lifecycle**: `DronePhase` component tracks per-drone state: `Idle` (hovering at start), `Racing` (following race spline), `Returning` (flying back to start), `Crashed` (destroyed, hidden). Transitions: `Idle → Racing` on countdown end, `Racing → Returning` when individual drone's `RaceProgress` marks it finished (validated by gate trigger detection) and `spline_t >= cycle_t + FINISH_EXTENSION`, `Returning → Idle` when drone reaches end of return spline, `Racing → Crashed` on gate miss (DNF). Crashed drones have `Visibility::Hidden`, zero velocity/angular velocity, and are skipped by all physics and AI systems. `ReturnPath` component (inserted on `Racing → Returning`, removed on `Returning → Idle`) holds a non-cyclic Catmull-Rom spline from the finish area back to start with per-drone variation. `drones_are_active()` run condition (replaces `race_is_running` for AI scheduling) returns true during `Racing` or when any drone is `Returning`, keeping AI systems alive during the post-race return flight. `hover_target` only writes to `Idle` drones.
- **Race leaderboard pattern**: `LeaderboardRoot` component marks an absolute-positioned panel (top-left, 190×~260px, semi-transparent `SMOKY_BLACK` bg, `GlobalZIndex(90)`). Spawned `OnEnter(Race)` with `DespawnOnExit(AppState::Race)`. 12 rows, each with `LbColorBar(pos)` (4px colored accent from `DRONE_COLORS`), `LbNameText(pos)` (position + name), `LbTimeText(pos)` (time/DNF). `update_leaderboard` reads `RaceProgress::standings()` each frame to reorder rows by race position. Visible only during `RacePhase::Racing` and `Finished`. Finished drones show `SEA_FOAM` time, crashed show `NEON_RED` "DNF", active show `SIDEWALK` elapsed. `DRONE_NAMES` and `DRONE_COLORS` are pub constants in `drone/spawning.rs`, shared by race UI, results UI, and spawning.
- **Race validation pattern**: `RacePhase` resource tracks the overall race state: `WaitingToStart → Countdown → Racing → Finished`. Click START RACE inserts `CountdownTimer` (3s). `tick_countdown` system decrements it; on expiry transitions to `Racing`, sets all drones to `DronePhase::Racing`, creates `RaceClock` and `RaceProgress`. Race logic runs as a 5-system `.chain()` in `Update`: `tick_countdown → tick_race_clock → gate_trigger_check → miss_detection → check_race_finished`. Gate trigger detection uses **plane-crossing**: `GatePlanes` resource (built once at race start from `GateIndex`/`GateForward`/`TriggerVolume` queries) caches each gate's plane (center, normal, right, up, half-extents). `gate_trigger_check` tests the line segment between `PreviousTranslation` and current `Transform` against the expected gate's plane — computes signed distances, requires front-to-back crossing (`d_prev > 0, d_curr <= 0`), interpolates the crossing point, then bounds-checks within `half_width * GATE_PASS_MARGIN` and `half_height * GATE_PASS_MARGIN` (margin = 1.1, 10% forgiveness). This eliminates tunneling at any speed and enforces directional validation (backwards passes rejected). `plane_crossing_check()` is a pure function with 8 unit tests. `point_in_trigger_volume()` is retained but no longer used for gate detection. Gate ordering enforcement: each drone tracks `next_gate` in `DroneRaceState`; after all gates, must pass gate 0 again (finish lap). `miss_detection` remains as a safety net — crashes drones (DNF: `DronePhase::Crashed`, hidden, zero velocity, explosion particles + sound) if `spline_t` advances past `(next_gate + 1) * POINTS_PER_GATE` without triggering the expected gate. `FINISH_EPSILON` (0.01) is added to the `spline_t >= finish_t` guards in `update_ai_targets` and `compute_racing_line` to prevent finish-line deadlock where `miss_detection`'s strict `>` could never fire. `update_ai_targets` in FixedUpdate checks `RaceProgress.finished` before allowing `Racing → Returning` transition, preventing a timing gap where the AI transitions before Update-side gate validation runs. `GatePlanes` resource is cleaned up `OnExit(Race)`. Resources (`RaceProgress`, `RaceClock`, `CountdownTimer`) are cleaned up `OnExit(Race)` and on race reset.
- **Course delete pattern**: `PendingCourseDelete` resource tracks a pending deletion. Clicking the "X" button on a course entry shows an inline confirmation (Yes/Cancel) inside the `ExistingCoursesContainer`. Confirming calls `delete_course()` and rebuilds the list. If the deleted course matches the currently loaded course name, the editor state is reset. If it matches `LastEditedCourse`, that resource is removed.
- **AI path following**: Each drone follows its own **unique cyclic Catmull-Rom spline** (`CubicCurve<Vec3>` from Bevy's `CubicCardinalSpline::new_catmull_rom().to_curve_cyclic()`) generated at spawn time by `generate_drone_race_path()`. The base path uses 3 control points per gate (approach, departure, midleg-to-next). Per-drone variation is applied to control points: `DroneConfig.gate_pass_offset` (fraction 0.2–0.4) shifts approach/departure points in 2D (width + height) at each gate using deterministic per-drone-per-gate Fibonacci hashing seeded by `RaceSeed` — offset is `fraction × trigger_volume_half_extents` so it scales with gate size, `DroneConfig.approach_offset_scale` (0.9–1.1) scales approach/departure distances, and `DroneConfig.racing_line_bias` (±4.4m, correlated with cornering aggression) shifts midleg waypoints laterally using deterministic per-drone-per-leg Fibonacci hashing (also seeded by `RaceSeed`). Gate forwards in `AIController` remain the true gate directions. Gate positions in per-drone `AIController` are offset in 2D (via `gate_pass_offset` × half-extents) so fallback distance checks match each drone's actual target point; the reference path `generate_race_path()` still uses true gate centers. `extract_sorted_gates()` returns gate half-extents (width, height from trigger volume, scaled by instance scale) alongside positions and forwards. The reference path `generate_race_path()` (no per-drone variation) is kept for the editor flight spline preview. `extract_sorted_gates()` is shared between both generators. Approach/departure waypoints use **adaptive offset**: `adaptive_approach_offset(gate_distance) = min(gate_distance * 0.3, 12.0)`. `POINTS_PER_GATE = 3.0` scales the spline parameter space: gate i midpoint is at `spline_t = i * 3 + 0.5`. The race is a **full lap**: drones start behind gate 0, fly through all gates, then back through gate 0 again (the finish line). `spline_t` extends past one full cycle (`cycle_t + FINISH_EXTENSION = gate_count * 3 + 1.5`) using `rem_euclid(cycle_t)` wrapping for spline sampling. Progress is updated via local tangent projection in `update_ai_targets()`, with a **minimum speed-based advancement** (`MIN_ADVANCE_SPEED_FRACTION = 0.25`) that prevents deadlocks when drones overshoot sharp turns and the tangent projection drops to zero. `compute_racing_line()` samples the spline at an **adaptive look-ahead** that shrinks on high-curvature sections (`CURVATURE_LOOK_AHEAD_SCALE = 30.0`, clamped to `[0.33, 1.0]` of base `LOOK_AHEAD_T`). A small organic noise wobble (0.3× amplitude) is applied laterally for natural movement. **Gate correction**: `compute_racing_line` blends `DesiredPosition` toward the per-drone gate offset position as the drone approaches each gate, simulating a pilot actively aiming for the opening. Uses parameter-space distance to the gate midpoint (`gate_i * 3.0 + 0.5`); correction starts `GATE_CORRECTION_RANGE` (2.0) units before the gate with quadratic ramp up to `GATE_CORRECTION_STRENGTH` (0.7) at the midpoint. Three effects: (1) target position lerps toward the drone's offset gate position, (2) velocity hint lerps toward the gate direction (so PID feedforward steers toward it), (3) organic wobble is suppressed proportional to blend. Handles the finish gate (gate 0 on second pass) via wrap-around. No correction during VictoryLap. **Curvature-aware speed limiting**: `DesiredPosition.max_speed` is set per-drone per-frame by scanning spline curvature over the next `SPEED_CURVATURE_RANGE * braking_distance` parameter units (5 samples, per-drone range), computing `v_safe = sqrt(lateral_accel * cornering_aggression / κ)` clamped to `[min_curvature_speed, max_speed]` (tunable via `AiTuningParams`). `apply_forces` respects this per-drone limit via soft exponential decay (rate 10.0/s, 99.5% converged within 0.5s) instead of hard-clamping velocity in a single frame. `RacePath` struct (in `paths.rs`) bundles the spline with gate positions and gate forwards. Requires >= 2 gates.
- **Dev dashboard pattern**: `AiTuningParams` resource (inserted via `init_resource`, persists across race restarts) holds 14 runtime-tunable AI/physics constants (including battery sag factor, dirty air strength, proximity avoidance radius/strength, velocity feedforward blend). All AI and physics systems read from this resource instead of `const` values. The dev dashboard UI (`drone/dev_dashboard.rs`) toggles with **F4** during Race state. Panel spawns top-right with per-parameter +/- buttons, value display (green when modified from default), and a Reset All button. Panel entity uses `DespawnOnExit(AppState::Race)`. Systems: `toggle_dev_dashboard`, `handle_param_buttons`, `handle_reset_button`, `update_param_labels`, `update_button_colors` (all in `Update`). `PARAM_META` const array stores display names, step sizes, and min/max ranges. `AiTuningParams::get(i)` / `set(i, value)` provide index-based access for the UI.
- **Gate directionality**: Each gate has a canonical forward direction. `TriggerVolumeConfig.forward` defines the local-space forward (default `Vec3::NEG_Z`). Per-instance `ObstacleInstance.gate_forward_flipped: bool` flips it 180 degrees. World-space forward = `rotation * (if flipped { -forward } else { forward })`. `GateForward` component (on gate entities at race time) stores the computed world-space forward. `RacePath.gate_forwards` and `AIController.gate_forwards` carry per-gate forward vectors for AI and validation. Gate forwards determine approach/departure waypoint placement in spline generation, so flipping a gate's direction changes the drone approach side. In the course editor, gates with `gate_order` show a **cyan arrow gizmo** pointing in the forward direction. Press `F` with a gate selected to flip its direction.
- **Aerodynamic perturbations**: `dirty_air_perturbation` system runs after `attitude_controller` and before `motor_lag` in the FixedUpdate chain. Two effects: (1) **Dirty air** — drones within 5m behind another drone (45° wake cone) receive angular velocity perturbation proportional to leader speed, producing visible wobble when drafting. Strength tunable via `AiTuningParams.dirty_air_strength` (default 4.0). (2) **Prop wash** — drones descending >2 m/s receive angular perturbation simulating flight through own downwash. Both use deterministic layered sine waves (no RNG) seeded by drone index. **Battery sag**: `apply_forces` linearly reduces effective thrust over race duration (`effective_thrust *= 1.0 - battery_sag_factor * race_progress`). Default 15% loss over 90s. Tunable via dev dashboard.
- **Proximity avoidance**: `proximity_avoidance` system runs after `compute_racing_line` and before `position_pid` in the FixedUpdate chain. Offsets `DesiredPosition` laterally when drones are nearby, so they veer around each other at full speed instead of slowing down from dirty-air-induced attitude instability. Algorithm: for each non-idle drone, scan all other non-idle drones within `avoidance_radius` (default 8m). Compute a lateral dodge vector perpendicular to own velocity (projects out velocity-aligned component). Head-on encounters use deterministic tiebreaker based on drone index comparison. Quadratic falloff `(1 - dist/radius)²` for smooth steering. Total offset scaled by `avoidance_strength` (default 12.0) and clamped to 3m max. **Gate-proximity suppression**: when a drone is within `GATE_SUPPRESS_RADIUS` (10m) of its next gate, avoidance offset is attenuated from 100% to `GATE_SUPPRESS_MIN` (20%), preventing drones from being pushed out of gate openings at the critical moment. O(12²) = 144 checks per tick, negligible. Both params tunable via dev dashboard.
- **Explosion effects**: `drone/explosion.rs` handles crash visual/audio effects with three layered particle types controlled by `ParticleKind` enum (field on `ExplosionParticle`). (1) **Debris** — 55 unlit emissive cubes in 3 size buckets (small/med/large), drone-colored, random outward velocities (upward-biased) plus 40% of the drone's crash velocity for directional scatter, gravity, linear scale shrink over 1.5s. (2) **HotSmoke** — 10 bright orange/red emissive cubes (0.5m), radial outward + upward drift with drag, grow-then-shrink scale (peaks at 25% of 2s lifetime). Gives the initial fiery look at impact. (3) **DarkSmoke** — 12 dark charcoal unlit cubes (0.8m), slower outward + upward drift with drag, grow-then-slowly-shrink scale using quadratic falloff (peaks at 20% of 3.5s lifetime, holds large then fades). Forms the dark halo around the hot core that lingers as grey smoke. All particles spawn centered at the point of impact. `ExplosionMeshes` resource pre-allocates mesh handles for all 5 particle sizes (debris small/med/large, hot smoke, dark smoke) once on `OnEnter(Race)` — `spawn_explosion` clones handles instead of allocating new meshes per explosion. `ExplosionSounds` resource holds 4 handles to `assets/sounds/explosion_{1..4}.wav`. Both loaded in `load_explosion_assets` on `OnEnter(Race)` and cleaned up `OnExit(Race)`. A random sound is chosen per explosion via `AudioPlayer` with `PlaybackSettings::DESPAWN`. If the sound files are missing, explosions are visual-only. Called from `miss_detection` in `race/gate.rs` which captures drone velocity before zeroing it.
- **Firework effects**: `drone/fireworks.rs` handles victory celebration when the first drone crosses the finish line. Two-layer effect: (1) **Gate confetti** — 30 small unlit emissive cubes in a horizontal fan, 50/50 mix of winner's drone color and gold (`SUNSHINE`), lifetime 1.5s with gravity (6.0 m/s²) and drag (2.0/s). Spawns instantly on first finish. (2) **Overhead shell bursts** — 3 staggered detonations at heights 15/20/25m above the origin point (delays 0.3/0.5/0.7s). Each shell spawns ~35 sparks (spherical distribution, gravity 4.0 m/s², drag 1.5/s, lifetime 2.0s, 70% drone color / 30% gold accent) and 4 willow particles (slower gravity 2.0 m/s², stronger drag 2.5/s, lifetime 3.0s, quadratic scale fade for lingering weep effect). `FireworkMeshes` resource pre-allocates 3 cuboid sizes (spark 0.06m, willow 0.1m, confetti 0.04m) on `OnEnter(Race)`. `FireworkSounds` loads `assets/sounds/firework.wav` (played per shell burst). `FireworksTriggered` marker resource prevents re-triggering. `PendingShell` components hold staggered detonation timers. `detect_first_finish` system reads `RaceProgress` to detect first finish; if `FireworkEmitter` entities exist (placed from course props), triggers effects at each emitter's position using its rotation for direction and optional `color_override` (falls back to winner's drone color); if no emitters exist, auto-fires at gate 0 (original behavior). All particles use `DespawnOnExit(AppState::Results)`. Resources cleaned up `OnExit(Results)`. ~180 particles total per emitter site, peak ~80 alive.
- **Course props pattern**: `CourseData.props` holds `Vec<PropInstance>`, each with `PropKind` (ConfettiEmitter or ShellBurstEmitter), translation, rotation, and optional `color_override` (`[f32; 4]` RGBA). `#[serde(default)]` ensures backward compatibility with existing `.course.ron` files. At race time, `spawn_course` creates invisible `FireworkEmitter` marker entities with `DespawnOnExit(AppState::Race)` from props. In the course editor, a tabbed UI (`EditorTab::Obstacles` / `EditorTab::Props`) switches between obstacle palette and prop palette. Props are represented by `PlacedProp` component (parallel to `PlacedObstacle`). `PlacedFilter = Or<(With<PlacedObstacle>, With<PlacedProp>)>` type alias is used in all shared gizmo/selection queries. Props render as CelMaterial cubes in the editor with sphere + directional gizmos (arrow for confetti, starburst for shell). Color override cycles through Auto (winner's color) / Gold / Red / Blue / Green / Purple / White. `PropEditorMeshes` resource holds shared mesh+material handles for prop placeholders.
- **Flight spline preview**: In CourseEditor mode, `draw_flight_spline_preview` builds a temporary `CourseData` from placed obstacles and calls `generate_race_path()` to produce the same cyclic Catmull-Rom spline used at race time. The spline is sampled every `0.1` parameter units and drawn as colored gizmo line segments. Color is based on curvature: **green** (drones can fly near max speed) → **red** (curvature forces drones to minimum speed). Uses `cyclic_curvature()` and `safe_speed_for_curvature()` from `drone::ai` (made pub for this purpose) with the live `AiTuningParams` resource. Requires >= 2 gates; draws nothing otherwise.
- **Camera switching pattern**: `CameraState` resource (persists globally, `init_resource`) holds `CameraMode` (Chase/Fpv/Spectator) and `target_standings_index` (for FPV). Default mode is `Chase` on `OnEnter(Race)`, reset to `Spectator` on `OnExit(Race)`. **Number keys** select mode: **1** = Chase, **2** = Spectator, **3** = FPV (repeated press cycles target drone in standings order). Each camera mode has its own update system gated by `camera_mode_is(mode)` run condition. **Chase camera** (`ChaseState` resource) follows the P1 (leader) drone closely — uses `RaceProgress::standings()` to find leader, blends in nearby drones within `PROXIMITY_RADIUS` (15m) for pack-racing framing (70% leader / 30% nearby). Constants: `CHASE_BEHIND=10`, `CHASE_HEIGHT=5`, `LOOK_AHEAD=5`. **FPV camera** (`FpvFollowState` resource) is a close stabilized follow cam ~2.5m behind and ~1.0m above the target drone. Extracts heading from horizontal velocity (yaw only, ignores roll/pitch wobble). Heavy position smoothing (5.0/s) and heading smoothing (4.0/s). Looks at drone + `LOOK_AHEAD_OFFSET` (2.0m ahead). **Spectator camera** (`SpectatorOrbitState` resource) uses RTS-style orbit controls matching the editor: middle-mouse drag to orbit, scroll to zoom, WASD for ground-plane movement. Orbit state initialized from current camera position on first use. Reuses `orbit_to_transform()` from `orbit.rs` and `CameraSettings` for sensitivity/zoom/speed. `CameraHudRoot` (bottom-left) shows current mode label and keybind hints. No cameras have collision — they pass freely through all geometry.
- **Results pattern**: `RaceResults` resource is built from `RaceProgress::to_race_results()` just before `Race → Results` state transition. Contains sorted `Vec<RaceResultEntry>` (drone_index, finished, finish_time, crashed, gates_passed), total_time, course_name. `ResultsTransitionTimer(0.5s)` is inserted when `RacePhase::Finished`, `tick_results_transition` counts down and triggers `NextState(Results)`. Results UI spawns `OnEnter(Results)` with `DespawnOnExit(AppState::Results)`: standings table (12 rows with color bars, names, times/DNF), "RACE AGAIN" button (`→ Race`), "MAIN MENU" button (`→ Menu`). `RaceResults` resource is cleaned up `OnExit(Results)`. `SelectedCourse` resource persists through Results so "RACE AGAIN" reloads the same course.

## Post-Phase Checklist

After completing each implementation phase:

1. **Update documentation**: Review and update `TODO.md`, `ARCHITECTURE.md`, and `CLAUDE.md` with any new types, patterns, or conventions introduced.
2. **Review warnings**: Run `cargo build` and review all warnings. Fix any that indicate real issues (unused imports, unnecessary mut, etc.). Warnings for types/functions that are planned for upcoming phases in the current sprint are acceptable and should be left alone — do not suppress them with `#[allow(dead_code)]`.
3. **Run tests**: Run `cargo test` and verify all tests pass. Add tests for new pure-logic functions.
4. **Manual testing feedback form**: After all automated checks pass, present a structured feedback form for manual testing. The form must include:
   - A checklist of every manually-testable behavior introduced or changed in the phase (specific actions, expected results).
   - Edge cases and error scenarios to verify (e.g., invalid input, rapid state transitions, boundary values).
   - Performance observations to watch for (frame drops, hitches, visual artifacts).
   - A "Pass / Fail / Notes" column for each item so the user can report results inline.
   - Regression checks: key existing behaviors that should still work unchanged.

   Format the form as a markdown table or checklist that can be filled out directly in chat.

## Testing Conventions

- Tests live in `#[cfg(test)] mod tests` at the bottom of each source file (idiomatic Rust).
- Use `tempfile` crate (dev-dependency) for filesystem tests — never write to the real `assets/` directory.
- Test pure logic and serialization (file I/O, data structures, discovery). ECS systems are tested manually.
- When adding file I/O functions, provide a parameterized version that accepts a `&Path` so tests can use temp directories (e.g., `discover_courses_in(path)` vs `discover_courses()`).
